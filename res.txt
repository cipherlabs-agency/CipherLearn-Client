  The Core Question: Multi-Tenant vs Multi-Instance                                                                                                                                 
  
  There are two fundamental approaches for selling CipherLearn to multiple tuition classes:                                                                                                                                                                                                                                                                             
  ---
  Approach 1: Multi-Instance (One deployment per client)

  Each client gets their own separate frontend, backend, and database.

  CipherLearn Portal (admin.cipherlearn.com)
      │
      ├── Client A: xyz-classes.cipherlearn.com
      │     ├── Frontend (Vercel)
      │     ├── Backend (Render/Railway)
      │     └── Database (Neon/Supabase)
      │
      ├── Client B: abc-academy.cipherlearn.com
      │     ├── Frontend (Vercel)
      │     ├── Backend (Render/Railway)
      │     └── Database (Neon/Supabase)
      │
      └── Client C: ...

  Pros: Complete data isolation, one bad client can't affect others, simple per-client customization, easy to reason about

  Cons: Expensive at scale, managing 50+ deployments is painful, updates must roll out to every instance

  ---
  Approach 2: Multi-Tenant Single Instance (Recommended)

  One deployment shared by all clients. Each client is a "tenant" identified by a tenantId in the database. Every table gets a tenantId column.

  CipherLearn Portal (admin.cipherlearn.com)
      │
      ├── Single Frontend (Vercel) ── app.cipherlearn.com
      ├── Single Backend (Render/Railway)
      └── Single Database (all tenants, row-level isolation)
            ├── tenant_1 rows
            ├── tenant_2 rows
            └── tenant_3 rows

  Pros: One deployment to manage, one codebase update affects all, cheapest by far, scales with a single infra

  Cons: More complex data isolation logic, need careful Prisma middleware to enforce tenant filtering, shared database risk (mitigated with proper design)

  ---
  What the CipherLearn Portal Needs

  A separate admin dashboard at admin.cipherlearn.com to manage all tenant instances:

  1. Tenant Management — Create/suspend/delete tenants, set class name, logo, branding
  2. Subscription & Billing — Track which tenants are active, plan tiers
  3. Health Monitoring — API uptime, database usage per tenant
  4. Deployment Pipeline — One-click updates to the shared codebase
  5. Custom Domain Mapping — Each client can optionally use their own domain

  ---
  What Changes in the Current Codebase (Multi-Tenant)

  The key change is adding a Tenant model and tenantId to every existing model:

  Database:
  - New Tenant table (id, name, slug, logo, domain, plan, isActive)
  - Add tenantId to: User, Student, Batch, Attendance, Assignment, Note, etc.
  - Prisma middleware to auto-filter by tenantId on every query

  Backend:
  - Tenant resolution middleware (from subdomain or custom domain)
  - Every query automatically scoped to current tenant
  - Super-admin routes for the CipherLearn Portal

  Frontend:
  - Tenant-aware theming (class name, logo from tenant config)
  - Subdomain-based routing (xyz.cipherlearn.com)
  - CipherLearn Portal as a separate Next.js app or route group

  ---
  My Recommendation

  Go with Multi-Tenant on the cheapest stack:

  1. Phase 1 (now): Add tenant model + config system to current codebase
  2. Phase 2: Build CipherLearn Portal for tenant management
  3. Phase 3: Move to production hosting (Vercel + Railway + Neon) — ~$25-54/month total
  4. Phase 4: Add custom domain support per tenant
  5. Phase 5: Billing integration (Razorpay/Stripe)

  This lets you sell to 50+ clients while paying under $100/month in hosting. Each new client is just a row in the Tenant table + a subdomain — no new deployments needed.
